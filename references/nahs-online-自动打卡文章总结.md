## 一、整体架构

分两阶段实现：
1. **逆向分析** — 反编译小程序、还原接口链路、破解签名与定位校验
2. **搭建服务** — CLI 工具 → FastAPI 后端 → 云服务器定时托管

## 二、逆向分析要点

### 2.1 获取小程序源码
- 学校打卡使用 uni-app 框架开发的微信小程序
- 通过反编译 `.wxapkg` 文件还原源码
- 关键文件：`request.js`（签名算法）、`authUtil.js`（OAuth 认证）、`stusign.js`（打卡交互逻辑）

### 2.2 关键参数获取
- **wechatAppId**：不在源码中明文出现，通过浏览器控制台注入 JS 脚本在运行时环境中递进扫描获取
- **OpenID**：学校后端获取接口已失效，只能通过代理抓包（Proxyman/Charles/Fiddler）或浏览器开发者工具从客户端侧截获

### 2.3 接口调用链
> 微信登录获取 openid → OAuth 换取 access_token → 查任务列表 → 查任务详情（含宿舍坐标）→ 查当日打卡状态 → 提交打卡

### 2.4 签名算法还原
- 每个业务请求带 `FlySource-sign` 请求头
- 计算规则：时间戳 + token 做 MD5 → 拼接 URL 路径再做 MD5 → 最终格式 `<MD5>1.<Base64时间戳>`
- 验证方法：同一组参数分别跑 JS 原版和自己的实现，输出一致即还原正确

### 2.5 GPS 定位校验绕过
- 任务详情接口直接返回宿舍精确经纬度、房间 ID 及允许精度上限
- 策略：以宿舍坐标为中心，每次打卡加随机小偏移（±0.0005 度，约几十米）
- 使用 Haversine 球面距离公式计算偏移距离，确认在允许范围内再提交

### 2.6 stuTaskId 校验字段
- 由客户端按固定规则自行生成，非服务器返回
- 将纬度、经度、精度、日期、任务 ID、文件 ID 六字段按特定顺序拼 JSON 后做 MD5
- Key 的顺序必须与 JS 源码中的对象字面量顺序完全一致

## 三、系统实现

### 3.1 CLI 工具
- 命令行交互工具，输入学号、密码、openid 即可操作
- 功能：查记录、立即打卡、看任务列表、按月查询打卡历史

### 3.2 后端服务
- **技术栈**：FastAPI + APScheduler + PostgreSQL
- 每天 21:00 通过 cron 触发器自动执行批量打卡
- 每个用户执行前加随机等待时间（0~5 分钟），分散请求防检测

### 3.3 凭据安全
- 密码：bcrypt 哈希存储（不可逆）+ MD5 哈希（学校接口要求）
- OpenID：AES-256-CBC 加密存储，密钥仅存于环境变量
- 接口鉴权：JWT Token
- 请求限流：Nginx 层面限制认证接口频率

### 3.4 数据库五张表
| 表 | 职责 |
|----|------|
| 用户表 | 账号、密码哈希、加密 openid、打卡开关、延迟参数 |
| 系统配置 | 全局键值对 |
| 任务缓存 | 缓存宿舍坐标等任务信息 |
| 打卡记录 | 每次打卡的日期、坐标、距离、结果 |
| 操作日志 | 每一步执行详情，方便排查 |

### 3.5 部署链路
> Nginx（反向代理 + 限流）→ Supervisor（守护进程）→ FastAPI（业务逻辑）→ PostgreSQL（数据存储）

一键部署脚本，自动完成装依赖、配数据库、建环境变量、配置守护和反向代理。

## 四、核心心得

1. **先跑通，再扩展** — 命令行几十行代码确认链路通了，再逐步加定时调度、数据库、部署脚本
2. **逆向靠耐心和交叉验证** — 同一组参数分别跑原版和自己的实现，输出一致即正确
3. **多层递进扫描** — 内存找不到去源码找，源码找不到去存储找，存储找不到就 hook 网络请求
4. **模拟真实行为** — 随机延迟、GPS 小幅偏移、请求频率控制，不给服务器造成压力
