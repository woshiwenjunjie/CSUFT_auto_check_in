## Architecture

**Purpose:** Automated dormitory check-in for Central South University of Forestry and Technology (õĖŁÕŹŚµ×ŚõĖÜń¦æµŖĆÕż¦ÕŁ”). Mimics a WeChat mini-program's network requests ŌĆö from OAuth login through GPS-simulated check-in submission ŌĆö via a reverse-engineered API client.

**Two-tier design:**
- **CLI tool** (`scripts/cli.py`, ~1000+ lines) ŌĆö interactive single-file terminal app. Used daily by end users for manual/one-off check-ins. Supports 10 subcommands with config persistence to `~/.auto_check_in/config.json`.
- **FastAPI server** (`src/main.py`) ŌĆö minimal scaffolding for phase 3 (multi-user web service). Not yet functional; only has a root route, CORS, and lifespan-managed `ApiClient` singleton.

**Core module ŌĆö `src/core/client.py` (`ApiClient`):**
The heart of the project. A stateful HTTP client wrapping all 8 school API endpoints across 3 domains:
1. **Auth** ŌĆö `get_captcha()`, `sign_in()` (password), `sign_in_openid()` (WeChat OpenID)
2. **Tasks** ŌĆö `get_task_list()`, `get_task_detail()` (returns dorm GPS coords + accuracy limit)
3. **Check-in** ŌĆö `get_one_record()`, `stu_sign()`, `stu_late_sign()`, `get_month_records()`

Every request carries three critical headers auto-generated by `_headers()`:
- `Authorization: Basic <base64(clientId:clientSecret)>` ŌĆö OAuth2 client credentials
- `FlySource-Auth: <token>` ŌĆö Bearer-style token from login
- `FlySource-sign: <signature>` ŌĆö custom HMAC-like signature (see below)
- `Referer` / `User-Agent` ŌĆö must impersonate WeChat mini-program environment, or the server rejects with "sign error"

**Supporting utils (`src/utils/`):**
- `crypto.py` ŌĆö `md5()`, `b64_encode()`, `b64_decode()` (thin wrappers over stdlib)
- `sign.py` ŌĆö `generate_sign(path, timestamp, token)` and `generate_basic_auth()`. Client credentials (`FLYSOURCE_CLIENT_ID`, `FLYSOURCE_CLIENT_SECRET`) read from env vars with hardcoded defaults for this specific school.
- `geo.py` ŌĆö `haversine()` distance calculation (meters) and `random_offset()` for GPS jitter

**Signature algorithm (`FlySource-sign`):**
```
sign = MD5(path + "?sign=" + MD5(timestamp + token)) + "1." + Base64(timestamp)
```
Where `path` is the URL path without query string, `timestamp` is a 13-digit millisecond epoch. The JS reference implementation at `scripts/sign.js` matches this exactly ŌĆö use it for cross-validation when debugging.

## Conventions

- **Always keep `AGENTS.md` in sync** ŌĆö update it when adding files, changing architecture, or fixing notable issues. Also append to `docs/CHANGELOG.md` and create a memory file in `docs/memory/` for significant decisions.
- **Cross-validate signing** ŌĆö when touching `src/utils/sign.py`, verify against `scripts/sign.js` with the same inputs. A one-character difference in the MD5 chain breaks all API calls.
- **WeChat headers are mandatory** ŌĆö the server returns misleading "sign error" when headers are missing. If `ApiClient` requests start failing, check `WX_USER_AGENT`, `Referer` (must use `servicewechat.com`), and `charset: utf-8` header before suspecting the signature algorithm.
- **Config priority:** env vars ŌåÆ hardcoded defaults in `ApiClient` class attributes. The CLI stores user-level config at `~/.auto_check_in/config.json` (token, username, openid, password, task_id, tenant_id).
- **Tests are fast and offline** ŌĆö only unit tests for utils (crypto, sign, geo). No integration tests against the real API. Keep it that way until phase 3 adds a test database.
- **Chinese comments and output** ŌĆö source code uses Chinese docstrings. CLI output is Chinese. Windows terminals need `sys.stdout.reconfigure(encoding='utf-8')` (already handled in `cli.py`).
- **All final responses must be in Chinese** ŌĆö every reply to the user should be written in Chinese (ń«ĆõĮōõĖŁµ¢ć). Code, commands, and technical identifiers remain in English, but explanations, summaries, and descriptions must be in Chinese.
