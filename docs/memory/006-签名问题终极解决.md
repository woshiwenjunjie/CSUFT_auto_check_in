# 006 — 签名问题终极解决

日期：2026-06-07

## 问题回顾

自 v0.4.0 以来，签名算法（`FlySource-sign` 请求头）始终阻塞项目进展。尽管从微信小程序 JS 源码中完整还原了算法逻辑，且 Python/Node.js 实现与源码一致，但所有需要签名的 API 请求都返回 `400 "签名错误"`。

此前排查过以下方向（均已排除）：
- MD5 编码方式（UTF-8 vs UTF-16LE vs Latin1）
- Base64 实现差异
- URL path 格式（相对路径 vs 完整 URL）
- timestamp 精度和格式
- token 拼接顺序
- 服务器时间同步

## 突破过程

### 第一步：抓包实时对比

用户通过 MuMu 模拟器 + Fiddler 正常运行小程序打卡，抓取了两条真实请求：

1. `GET /api/flySource-yxgl/dormSignRecord/getOne` — 签名 `b693ebaa465d8577827ec74b91689f371.MTc4MDgzODMzMDU5MA==`
2. `POST /api/flySource-yxgl/dormSignRecord/stuSign` — 签名 `57d195707732b50131955e1663eaca511.MTc4MDgzODMzMDU0MQ==`

### 第二步：重建签名验证算法

用 Python hashlib 以**完全相同的输入**（相同 timestamp、token、path）重新计算签名：

```
真实签名:   b693ebaa465d8577827ec74b91689f371.MTc4MDgzODMzMDU5MA==
Python 输出: b693ebaa465d8577827ec74b91689f371.MTc4MDgzODMzMDU5MA==
✅ 逐字节完全匹配！
```

两条签名均完美复现，**确认 MD5/Base64 实现与小程序完全一致**。

### 第三步：找到真正原因

既然算法正确，为什么 Python 新生成的请求仍然返回 400？

对比 Fiddler 抓包和 Python httpx 请求的**完整 HTTP 头**：

| 请求头 | Python (旧) | 微信小程序 |
|--------|-------------|-----------|
| `charset` | ❌ 缺失 | `utf-8` |
| `Referer` | ❌ 缺失 | `https://servicewechat.com/wx0e47c34c9982aa09/7/page-frame.html` |
| `User-Agent` | ❌ 缺失 | `Mozilla/5.0 ... MicroMessenger/8.0.72 ... MiniProgramEnv/android` |

### 第四步：验证假设

```python
# 不加微信头 → 400 "签名错误"
headers_bare = {"Authorization": ..., "FlySource-sign": ...}
r = httpx.get(url, headers=headers_bare)  # → 400

# 加上微信头 → 200 OK ✅
headers_wx = {**headers_bare, "Referer": "...", "User-Agent": "...", "charset": "utf-8"}
r = httpx.get(url, headers=headers_wx)  # → 200 OK!
```

## 结论

**签名算法自始至终是正确的。**

服务器真正的校验逻辑是：
1. 验证 `FlySource-sign` 签名（算法正确即可通过）
2. **验证 `Referer` 和 `User-Agent` 请求头**（必须包含微信小程序特征）
3. 两者缺一不可，缺少微信头直接返回"签名错误"（服务端的统一反爬策略）

```python
    WX_VERSION = "7"
    WX_USER_AGENT = "Mozilla/5.0 ... MicroMessenger/8.0.72 ... MiniProgramEnv/android"

    def __init__(self, ...):
        self._referer = f"https://servicewechat.com/{self.WX_APP_ID}/{self.WX_VERSION}/page-frame.html"

        # ...
```

## 经验教训

1. **"签名错误"不一定是算法问题**：服务器可能在校验其他层面的合法性（反爬、环境检测）
2. **抓包对比是关键**：不要只看 JS 源码，要对比完整的 HTTP 请求头和请求体
3. **逐字节验证**：用真实抓包数据中的 timestamp+token 重新计算签名，可以彻底确认/排除算法问题
4. **微信小程序接口保护**：Referer 和 User-Agent 是常见的非小程序拦截手段，必须伪装
