# 014 — UTC / 北京时间时间坑

**日期**：2026-06-09

## 问题

flySource API 服务器的打卡窗口判定基于 **UTC 时间**，而非北京时间（UTC+8）。多次在此踩坑：

1. **误以为服务器返回的时间是北京时间** — `signStartTime`/`signEndTime` 是 UTC 时间
2. **本地调试成功但自动打卡失败** — 本地刚好在北京时间窗口内（21:00–22:30），但服务器判定窗口实际是 UTC 13:00–14:30
3. **通知文案显示北京时间误导排错** — "窗口 21:00–22:30" 让人以为失败原因在别处

## 根因

- 服务器系统时区为 UTC，所有时间比较直接取 `datetime.utcnow()`
- 小程序/Web 端显示时做了 `UTC+8` 转换，但 API 直调用不会做转换
- `signStartTime`/`signEndTime` 字段值是 UTC 时刻（如 `"13:00"` 对应北京时间 21:00）

## 关键数字

| 概念 | UTC | 北京时间 |
|------|-----|---------|
| 打卡窗口 | 13:00–14:30 | 21:00–22:30 |
| GitHub Actions 定时 | 13:05 | 21:05 |

## 对应代码

```python
# scripts/__init__.py 中 datetime.now() 已在 v0.8.1 改为硬编码 UTC+8
# 但服务器端比较用的是 UTC，本地只需确保请求时间在 UTC 13:00–14:30
```

## 教训

- **服务器时间总是 UTC**，不要假设任何 API 返回的时间字段是本地时间
- 显示给用户时需要主动做 `UTC+8` 转换
- GitHub Actions cron 表达式中的时间是 UTC（`5 13 * * *` = UTC 13:05）
- ~~通知文案中窗口时间优先标注 UTC~~（v0.8.3 修正：用户只看北京时间，UTC 标注反而造成困惑）
- **v0.8.3 修正**：通知时间统一用北京时间，通过 Python `datetime.now(timezone(timedelta(hours=8)))` 获取
- **bash `date` + `TZ` 不可靠**，全部时间值必须用 Python 显式时区（纯数学偏移，不依赖系统）
- **不要设 `TZ` workflow 环境变量**，可能影响 cron 调度器
