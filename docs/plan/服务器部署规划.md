## 2. 多用户 OpenID 体系

### 2.1 为什么每个用户需要自己的 OpenID

OpenID 是**微信用户 + 小程序**维度的唯一标识：

```
微信用户 (UnionID)
  └── 小程序 A → OpenID_A
  └── 小程序 B → OpenID_B
```

- 用户 A 在小程序中的 OpenID 是 `oABC...`
- 用户 B 在小程序中的 OpenID 是 `oXYZ...`
- 学校的 OAuth 接口用 `OpenID + 学号 + 密码` 换取 `access_token`

**所以每新增一个用户，必须获取其专属的 OpenID**。

### 2.2 新用户入驻流程

```
┌───────────────────────────────────────────────┐
│  第一步：管理员收集用户信息                       │
│  - 学号、姓名（用户提供）                         │
│  - 宿舍坐标（从学校 API 自动获取）                 │
├───────────────────────────────────────────────┤
│  第二步：用户提供 OpenID（一次性操作）              │
│                                               │
│  方案 A：用户自行抓包                           │
│    用户按照教程 Fiddler + 模拟器抓包              │
│    → 提交 OpenID 给管理员                       │
│                                               │
│  方案 B：管理员统一抓包                          │
│    用户提供微信临时登录权限                       │
│    → 管理员模拟登录抓包获取 OpenID                │
│                                               │
│  方案 C：自建代理服务器（自动化）                  │
│    部署 HTTPS 代理 + 证书注入                    │
│    → 用户连接代理后自动捕获 OpenID               │
├───────────────────────────────────────────────┤
│  第三步：系统绑定                                │
│    管理员在后台执行：                            │
│    POST /api/users/register                   │
│    {                                          │
│      "student_id": "2023****",                │
│      "openid": "o************...",                │
│      "password": "加密传输"                     │
│    }                                          │
│    → 系统自动调用 OAuth 获取 token               │
│    → 存储加密凭据到数据库                        │
├───────────────────────────────────────────────┤
│  第四步：自动托管                                │
│    APScheduler 每天 21:05 自动打卡               │
│    → 打卡结果推送（微信通知 / 邮件 / Webhook）    │
└───────────────────────────────────────────────┘
```

### 2.3 数据隔离

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ User A       │     │ User B       │     │ User C       │
│ openid: oA.. │     │ openid: oB.. │     │ openid: oC.. │
│ token: jwtA  │     │ token: jwtB  │     │ token: jwtC  │
│ dorm: 坐标A   │     │ dorm: 坐标B   │     │ dorm: 坐标C   │
│ schedule: 21h │     │ schedule: 21h │     │ schedule: 21h │
└──────────────┘     └──────────────┘     └──────────────┘
       │                     │                     │
       ▼                     ▼                     ▼
  ApiClient_A           ApiClient_B           ApiClient_C
  (独立 httpx)          (独立 httpx)          (独立 httpx)
```

每个用户对应独立的：
- `ApiClient` 实例（含独立 token 和 httpx 连接池）
- 定时任务 Job（独立调度，互不影响）
- 数据库中的凭据（AES-256-CBC 加密存储）

## 3. 技术栈选型

| 层 | 技术 | 说明 |
|----|------|------|
| Web 框架 | FastAPI | 异步、自动 OpenAPI 文档、类型安全 |
| ORM | SQLAlchemy 2.0 | 异步支持、迁移工具 Alembic |
| 数据库 | PostgreSQL 15 | 生产级、JSON 支持、连接池 |
| 定时任务 | APScheduler | Python 原生、支持持久化到 DB |
| 任务队列 | Celery + Redis | 批量打卡、异步通知（规模扩展时） |
| 反向代理 | Nginx | SSL 终结、静态资源、限流 |
| 进程管理 | Supervisor / systemd | 守护进程、自动重启 |
| 容器化 | Docker + Compose | 可选的简化部署方案 |
| 凭据加密 | AES-256-CBC | 数据库存储前加密 |
| 日志 | structlog + Loki | 结构化日志、聚合查询 |

**最小可行部署**（10 人以下）：

```
FastAPI + SQLite + APScheduler + uvicorn
单进程，无需 Nginx/Supervisor/PostgreSQL
Cron 触发即可，甚至可纯 CLI + cron
```

## 4. 数据库设计

### 4.1 核心表

```sql
-- 用户表
CREATE TABLE users (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    student_id   VARCHAR(20)  NOT NULL UNIQUE,     -- 学号
    tenant_id    VARCHAR(10)  DEFAULT '000000',     -- 租户 ID
    openid       VARCHAR(64)  NOT NULL,             -- 微信 OpenID
    openid_enc   BYTEA,                             -- AES 加密的 OpenID
    password_enc BYTEA        NOT NULL,             -- AES 加密的密码
    dorm_lat     DOUBLE PRECISION,                  -- 宿舍纬度
    dorm_lng     DOUBLE PRECISION,                  -- 宿舍经度
    dorm_name    VARCHAR(100),                      -- 宿舍楼名
    room_no      VARCHAR(20),                       -- 房间号
    is_active    BOOLEAN      DEFAULT TRUE,         -- 是否启用
    created_at   TIMESTAMP    DEFAULT NOW(),
    updated_at   TIMESTAMP    DEFAULT NOW()
);

-- Token 表（记录 token 生命周期）
CREATE TABLE tokens (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID REFERENCES users(id) ON DELETE CASCADE,
    access_token TEXT         NOT NULL,             -- JWT token
    expires_at   TIMESTAMP    NOT NULL,             -- 过期时间
    created_at   TIMESTAMP    DEFAULT NOW(),
    UNIQUE(user_id)  -- 每个用户只有一条当前 token
);

-- 打卡任务表（缓存学校 API 的任务信息）
CREATE TABLE tasks (
    task_id      VARCHAR(64)  PRIMARY KEY,          -- 学校 API 的 taskId
    user_id      UUID REFERENCES users(id),
    task_name    VARCHAR(100),
    sign_start   TIME,
    sign_end     TIME,
    location_acc INTEGER      DEFAULT 100,          -- 精度上限（米）
    is_active    BOOLEAN      DEFAULT TRUE,
    updated_at   TIMESTAMP    DEFAULT NOW()
);

-- 打卡记录表
CREATE TABLE checkin_records (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID REFERENCES users(id),
    task_id      VARCHAR(64),
    sign_date    DATE          NOT NULL,
    sign_time    TIME,
    sign_lat     DOUBLE PRECISION,
    sign_lng     DOUBLE PRECISION,
    sign_status  INTEGER       DEFAULT 0,           -- 0=正常
    is_late      BOOLEAN       DEFAULT FALSE,       -- 是否补签
    raw_response JSONB,                              -- API 原始响应
    created_at   TIMESTAMP     DEFAULT NOW(),
    UNIQUE(user_id, task_id, sign_date)
);

-- 操作日志表
CREATE TABLE logs (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID REFERENCES users(id),
    level        VARCHAR(10)   DEFAULT 'INFO',
    action       VARCHAR(50)   NOT NULL,             -- login/checkin/error
    detail       TEXT,
    created_at   TIMESTAMP     DEFAULT NOW()
);

-- 定时任务配置表
CREATE TABLE schedules (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      UUID REFERENCES users(id) UNIQUE,
    task_id      VARCHAR(64),                        -- 关联的任务 ID
    cron_hour    INTEGER       DEFAULT 21,           -- 打卡时间（时）
    cron_minute  INTEGER       DEFAULT 5,            -- 打卡时间（分）
    timezone     VARCHAR(50)   DEFAULT 'Asia/Shanghai',
    is_active    BOOLEAN       DEFAULT TRUE,
    created_at   TIMESTAMP     DEFAULT NOW()
);
```

### 4.2 ER 关系图

```
users (1)────(1) tokens         -- 每个用户一个活跃 token
users (1)────(N) tasks           -- 每个用户有多个任务（通常 1-2 个）

users (1)────(1) schedules       -- 打卡时间配置
```

## 6. 定时任务设计

### 6.1 APScheduler 架构

```python
# 启动时加载所有活跃用户的定时任务
async def init_scheduler():
    scheduler = AsyncIOScheduler()
    jobstores = {"default": SQLAlchemyJobStore(url=DATABASE_URL)}
    
    users = await get_active_users()
    for user in users:
        schedule = await get_schedule(user.id)
        scheduler.add_job(
            func=auto_checkin,
            trigger=CronTrigger(
                hour=schedule.cron_hour,
                minute=schedule.cron_minute
            ),
            args=[user.id],
            id=f"checkin_{user.id}",
            replace_existing=True,
        )
    
    scheduler.start()
```

### 6.2 打卡任务执行流程

```python
async def auto_checkin(user_id: str):
    """自动打卡任务"""
    user = await get_user(user_id)
    if not user.is_active:
        return
    
    # 1. 检查 token 是否即将过期（< 3 天）
    token = await get_token(user_id)
    if token.expires_in_days() < 3:
        await refresh_token(user)
    
    # 2. 获取任务列表
    client = ApiClient(token.access_token)
    tasks = client.get_task_list()
    
    # 3. 逐个任务打卡
    for task in tasks:
        # 3.1 先查是否已打卡
        record = client.get_one_record(task.task_id)
        if record.already_checked_in:
            logger.info(f"User {user.student_id}: already checked in for {task.task_name}")
            continue
        
        # 3.2 获取宿舍坐标
        detail = client.get_task_detail(task.task_id)
        dorm = detail.dormitory
        
        # 3.3 生成模拟 GPS
        lat, lng = random_offset(dorm.lat, dorm.lng)
        
        # 3.4 距离校验
        dist = haversine(lat, lng, dorm.lat, dorm.lng)
        if dist > detail.location_accuracy:
            logger.warning(f"User {user.student_id}: GPS out of range, retrying")
            lat, lng = random_offset(dorm.lat, dorm.lng, max_offset=0.0002)
        
        # 3.5 提交打卡
        resp = client.stu_sign({...})
        
        # 3.6 记录结果
        await save_checkin_record(user.id, task.task_id, resp)
        await save_log(user.id, "checkin", str(resp))
        
        # 3.7 随机延迟（模拟真实行为，避免同时请求）
        await asyncio.sleep(random.randint(5, 30))
```

### 6.3 Token 自动刷新

```
┌──────────────────────────────────────┐
│  每天 12:00 检查所有用户 token        │
│                                      │
│  expires_in < 3 天 → 自动刷新        │
│  刷新逻辑：                           │
│    1. 读取加密的 openid + password    │
│    2. 解密                            │
│    3. 调用 sign_in_openid()           │
│    4. 更新 tokens 表                  │
│    5. 记录日志                        │
│                                      │
│  刷新失败 → 通知管理员                │
└──────────────────────────────────────┘
```

## 8. 安全设计

### 8.1 凭据加密

```
┌─────────────────────────────────────────┐
│  存储层加密                               │
│                                         │
│  password      → AES-256-CBC → BYTEA   │
│  openid        → AES-256-CBC → BYTEA   │
│  encryption_key → 环境变量，不入代码库    │
│                                         │
│  传输层加密                               │
│  API 全走 HTTPS (Nginx SSL 终结)         │
│  管理接口验证 X-Admin-Key               │
│  JWT 用户验证 + RBAC 角色控制            │
└─────────────────────────────────────────┘
```

### 8.2 安全检查清单

- [ ] `.env` 不入 git（`.gitignore` 已配置）
- [ ] 数据库凭据不硬编码
- [ ] `ENCRYPTION_KEY` 最小 32 字节
- [ ] Admin API 限流（每 IP 5 次/分钟）
- [ ] 打卡失败自动重试 + 通知
- [ ] 日志脱敏（不记录完整密码/OpenID）
- [ ] PostgreSQL 使用 SSL 连接
- [ ] 定期备份数据库（`pg_dump` + cron）
