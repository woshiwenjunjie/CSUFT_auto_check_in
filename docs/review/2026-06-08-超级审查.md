## 二、🔴 关键问题（Critical）

### C1 — `month` 命令参数顺序错误

**文件**: `scripts/cli.py` 行 1289–1292
**严重程度**: 🔴 Critical — 文档中的用法不可用

```python
p_month.add_argument("task_id", nargs="?", default="", ...)
p_month.add_argument("month", ...)
```

`task_id` 是第一个 positional，`month` 是第二个 positional（required）。

实际行为：
- `cli.py month 2026-06` → `task_id="2026-06"`, `month` 缺失 → **argparse 报错**
- `cli.py month "" 2026-06` → 可用但没有人会这样用
- `cli.py month <real_id> 2026-06` → 可用但绕过了自动保存机制

**但文档 epilog 写的正是** `python scripts/cli.py month 2026-06`，用户按文档操作会直接报错。

**修复**：把 `task_id` 改为 `--task-id` 可选参数，`month` 作为唯一的 positional：
```python
p_month.add_argument("--task-id", default="", help="任务 ID，留空自动从配置读取")
p_month.add_argument("month", help="月份，格式 YYYY-mm（如 2026-06）")
```

同步更新 `docs/guides/user/CLI教程.md` 中所有 `month` 示例。

### C2 — ApiClient 资源泄漏（CLI 中从不关闭）

**文件**: `scripts/cli.py` 行 268–271, `src/core/client.py` 行 51–53
**严重程度**: 🔴 Critical — httpx 连接池永不释放

```python
def get_client():
    """Build an ApiClient from saved token, returning (client, config_dict)."""
    cfg = load_config()
    return ApiClient(cfg.get("token", "")), cfg
```

`ApiClient.__init__` 创建 `httpx.Client(...)`，但 CLI 所有命令都通过 `get_client()` 获取客户端后，**从未调用 `.close()`**。每次运行 CLI 命令，httpx 连接池残留直到进程退出。对于单次命令影响不大，但如果未来 CLI 改为 REPL 模式或多用户循环调用，连接会累积。

**修复**：两种方案——
1. **(推荐)** 在 CLI 中使用 context manager：让 `get_client` 返回 context manager，所有命令用 `with` 包裹
2. **(最小改动)** 注册 `atexit` handler：`atexit.register(client.close)`

## 三、🟡 重要问题（Important）

### I1 — `--lat` / `--lng` 只传一个时静默忽略

**文件**: `scripts/cli.py` 行 903–907
**严重程度**: 🟡 — 用户传了参数但没生效，结果与预期不符

```python
if args.lat is not None and args.lng is not None:
    cur_lat, cur_lng = float(args.lat), float(args.lng)
    source = "手动指定"
else:
    cur_lat, cur_lng = random_offset(dorm_lat, dorm_lng, args.offset)
    source = f"模拟偏移 ±{args.offset}°"
```

如果用户只传了 `--lat 28.13` 而忘记传 `--lng`，代码静默 fallback 到随机偏移，用户以为坐标已指定但实际没有。应该在只有一个传了另一个没传时报错退出。

**修复**：
```python
if args.lat is not None and args.lng is not None:
    ...
elif args.lat is not None or args.lng is not None:
    print(c(Style.error, "  --lat 和 --lng 必须同时指定"))
    return
else:
    ...
```

### I2 — `auto_checkin.sh` STATUS_RAW 提取逻辑脆弱

**文件**: `scripts/auto_checkin.sh` 行 187–188
**严重程度**: 🟡 — 通知中的状态字段经常显示"未知"

```bash
STATUS_RAW=$(extract_field "$CHECKIN_OUTPUT" "状态")
STATUS_RAW="${STATUS_RAW:-未知}"
```

依赖链：`cmd_checkin` 输出 → 打卡后回查 `get_one_record` → `kv("状态", _status_display(sc, sn))` → `sed -n "s/.*状态:\s*//p"`。

**脆弱点**：
1. 如果回查 token 过期 → 函数提前 return → 没有"状态:"行 → STATUS_RAW="未知"
2. 如果回查 API 失败 → 没有"状态:"行 → STATUS_RAW="未知"
3. ANSI 模式下（本地运行），提取的状态值带颜色码 → 通知内容可能异常

**修复**：在 `cmd_checkin` 中显式输出一个机器可读的标记行，bash 脚本用该行解析：
```python
# cli.py 中增加（颜色始终关闭）
print(f"STATUS:{sign_status_code}:{sign_status_name}")
```
```bash
# bash 中用更精确的匹配
STATUS_CODE=$(echo "$CHECKIN_OUTPUT" | grep -oP 'STATUS:\d+:' | cut -d: -f2)
```

### M2 — `random_offset` 污染全局随机状态

**文件**: `src/utils/geo.py` 行 24
**严重程度**: 🟢 — 对当前 CLI 场景无影响，但不符合最佳实践

```python
rng = random.Random(seed) if seed is not None else random
```

当 `seed=None` 时，直接使用 `random` 模块（模块级单例），调用 `random.uniform()` 会改变全局随机状态。虽然 CLI 是短生命周期进程，不会有影响，但如果在阶段三的多用户服务中共享此模块，可能导致随机性偏差。

**建议**: 总是创建独立实例：
```python
rng = random.Random(seed)  # seed=None 时自动使用系统熵源
```

## 五、🔵 观察与建议（Observations）

### O1 — 测试覆盖缺口

当前 20 个测试全部针对纯函数（crypto/sign/geo），以下模块完全没有测试：

| 模块 | 行数 | 风险 |
|------|------|------|
| `ApiClient` (client.py) | 196 | 高 — 所有 API 调用的核心 |
| CLI 命令 (cli.py) | 1356 | 中 — 10 个命令，用户直接使用 |
| `auto_checkin.sh` | 316 | 高 — 生产环境每日运行 |

阶段三应优先补齐 `ApiClient` 的 mock 测试（用 `responses` 或 `pytest-httpx`）。

---

### O2 — 无日志系统

当前所有输出用 `print()`，无可追溯的持久化日志。阶段三需引入 `logging` 模块：
- `logging.getLogger("auto_checkin")` 统一日志器
- FileHandler 写入 `~/.auto_check_in/checkin.log`
- GitHub Actions 中 `LOG_FILE` 已存在（`/tmp/auto_checkin_output.txt`），但 Python 侧没有写日志

---

### O3 — CLI 错误退出码缺失

`cmd_checkin` 失败时所有路径都只 `return`，不设退出码。GitHub Actions 的 `set -e` 不会因 Python 返回非零退出码而中止——但如果 `main()` 返回非零，Actions 可以据此判断成功/失败。当前 `sys.exit()` 仅出现在 `KeyboardInterrupt` 间接处理中。

**建议**: 各命令失败路径使用 `sys.exit(1)`。

---

### O4 — `config clear` 操作无备份

**文件**: `scripts/cli.py` 行 552–553

```python
if CONFIG_FILE.exists():
    CONFIG_FILE.unlink()
```

直接删除文件，没有任何备份或回收机制。误操作后用户需要重跑 `setup` 且重新抓包获取 OpenID。

**建议**: 删除前备份到 `config.json.bak`，或至少显式打印将被删除的内容。

---

### O5 — `.keepalive` 文件无 `.gitignore` 条目

`.keepalive` 是保活机制故意提交的文件，不在 `.gitignore` 中。但如果用户在本地运行 workflow（`act`），本地也会创建这个文件。建议在 `.gitignore` 中加注释说明为什么此文件**不能**被 ignore。

## 九、结论

项目整体质量**良好**。核心逆向算法、API 调用链、GitHub Actions 自动化均已稳定运行。前三次审查修复了所有已知的 P0/P1 问题。

本次超级审查新发现 **2 个 Critical**（month 命令 bug、连接池泄漏）和 **4 个 Important**（参数校验、通知字段脆弱性、凭据泄漏、依赖膨胀），均为此前审查未覆盖的深层问题。建议在 v0.8.2 中集中修复，然后进入阶段三开发。

**阶段三阻塞项**：C2（连接池泄漏）需在多用户服务开始前修复，其余可在阶段三进行中逐步处理。
