## 一、本次修复清单

### ✅ P1 — `cmd_status` 已打卡却显示"未打卡"（三重根因）

| # | 根因 | 修复 |
|---|------|------|
| 1 | `get_one_record(task_id)` 未传日期，API 可能不返回当天记录 | 显式传入 `datetime.now().strftime("%Y-%m-%d")` |
| 2 | 缺少 `_token_expired` 检查，token 过期被误报为"未打卡" | 添加 `_token_expired(rec)` 检测，明确提示"Token 已过期" |
| 3 | 判断 `rec.get("data")` 在 `data: {}` 时因空字典 falsy 错误落入"未打卡" | 改为 `d and d.get("signStatus") is not None` 精确判断 |

### ✅ P1 — `status_map` 字典重复 3 次

提取为模块级常量 `STATUS_MAP: dict[int, str]` + 颜色编码函数 `_status_display(sc, sn)`。`cmd_checkin`、`cmd_record`、`cmd_month` 全部统一使用。

### ✅ P1 — `cmd_status` 状态颜色编码不完整

从子串匹配 `"正常" in sn or "已打卡" in sn` → 统一使用 `_status_display(sc, sn)`，覆盖全部 7 种状态。

### ✅ P2 — `cmd_checkin` 确认查询缺少 token 过期检测

已添加 `_token_expired(record_resp)` 检测。

### ✅ P3 — `_mask` 调用点冗余

`_mask` 改为 None-safe（返回 `""`），调用简化为 `_mask(cfg.get("xxx"), N) or "fallback"`，消除 4 处 `if cfg.get("xxx") else` 冗余守卫。

### ✅ P3 — `cmd_login` 缺失 masked 输出

登录成功补充 `kv("学号", _mask(username, 3))` + `kv("Token", _mask(...))`。

### ✅ P3 — argparse 死参数 `--token`

`config clear --token` 已移除（默认行为就是清除 token，`--token` 未在 handler 中使用）。

## 二、深度审查：简洁性

### 当前评分：★★★★☆

CLI 单文件约 1230 行，结构清晰：

```
L1-39    导入 + Windows UTF-8 修复
L42-130  Style 终端系统 + 输出辅助（divider/kv/bullet/warn_box）
L132-170 Spinner 动画
L173-191 配置持久化
L194-258 secure_input 跨平台密码输入
L261-325 共享工具（get_client / _resolve_task_id / _token_expired / _mask / STATUS_MAP / _status_display）
L330-395 cmd_setup
L398-515 cmd_status
L518-580 cmd_config
L582-646 cmd_login_openid
L648-688 cmd_login
L692-744 cmd_tasks
L746-810 cmd_detail
L814-837 _prepare_stu_sign_data
L839-966 cmd_checkin
L968-1025 cmd_record
L1028-1089 cmd_month
L1092-1183 _build_parser + _show_welcome + main
```

**优点：**
- 每个函数职责单一，行数适中（最长 cmd_checkin ~130 行，含大量 kv 输出）
- 共享逻辑正确提取：`get_client`、`_resolve_task_id`、`_token_expired`、`_mask`、`_status_display`
- 终端输出辅助函数统一：`divider`、`kv`、`bullet`、`warn_box`
- 错误处理一致：每个命令都有 token 过期 → 提示重新登录 → return 的标准模式

**可简化的点（已评估，保留现状）：**

| 候选 | 决策 | 理由 |
|------|------|------|
| `cmd_status` / `cmd_config` 的凭据展示（~8 行重复） | 不提取 | 上下文不同（status 是概览，config 是详情），提取后反而增加认知负担 |
| `cmd_login_openid` / `cmd_login` 登录成功展示（~12 行重复） | 不提取 | `cmd_login` 是备用路径，逻辑略有差异（无 "下一步" 提示），过度抽象降低可读性 |
| `cmd_status` / `cmd_record` 的记录展示（~10 行重复） | 不提取 | 语义不同（"今日尚未打卡" vs "暂无打卡记录"），合并会引入参数复杂度 |

## 三、深度审查：可扩展性

### 当前评分：★★★☆☆

**已具备的扩展点：**

| 扩展点 | 机制 | 覆盖 |
|--------|------|------|
| API 端点 | `ApiClient` 类方法 — 新增端点只需加一个方法 | ✅ 良好 |
| 签名凭据 | 环境变量 `FLYSOURCE_CLIENT_ID/SECRET` 覆盖 | ✅ 良好 |
| API 基址 | 环境变量 `CHECKIN_BASE_URL` + 构造函数参数 | ✅ 良好 |
| 微信伪装头 | 环境变量 `WX_APP_ID/VERSION/USER_AGENT` 覆盖 | ✅ 良好 |
| CLI 子命令 | `dispatch` dict → 添加 `cmd_xxx` + parser 即可 | ✅ 良好 |
| 状态码映射 | `STATUS_MAP` 模块级常量，新增状态码只需加一行 | ✅ 已改进 |
| 颜色主题 | `Style` 类，新增样式加一个 classmethod | ✅ 良好 |

**限制（阶段三需处理）：**

| 限制 | 说明 | 建议 |
|------|------|------|
| CLI 单文件 | 1230 行，随着功能增长需要拆分 | 阶段三拆为 `cli/` 包：`display.py`、`commands/`、`config.py` |
| 配置键硬编码 | `"token"`、`"username"` 等散落各处 | 提取 `CONFIG_KEYS` 常量类 |
| 多用户支持 | 当前 cfg 为单用户设计，无用户 ID 概念 | 阶段三引入 user_id 作为 cfg key |
| 无日志系统 | `print()` 直接输出，无可追溯日志 | 阶段三引入 `logging` 模块 + 文件 handler |
