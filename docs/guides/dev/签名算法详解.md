## 1. 为什么需要签名

学校 API 每个请求都必须携带 `FlySource-sign` 请求头和 `Authorization: Basic ...` 认证头。缺少任何一个都会导致请求被拒。

签名的作用：
- **防伪造**：没有正确的 `ClientSecret` 和 `access_token`，无法生成合法签名
- **防重放**：签名中包含毫秒级时间戳，服务器可检测重放攻击
- **完整性校验**：签名覆盖请求路径，路径被篡改则签名不匹配

## 2. FlySource-sign 算法

### 公式

```
sign = outer_md5 + "1." + base64(timestamp)
```

其中：

```
outer_md5 = MD5(path + "?sign=" + inner_md5)
inner_md5 = MD5(timestamp_str + token)
```

### 参数说明

| 参数 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `path` | string | URL 路径 | 去掉 `?query` 部分，如 `/api/.../getListForApp` |
| `timestamp` | int | `datetime.now()` | **13 位**毫秒 Unix 时间戳 |
| `token` | string | OAuth 响应 | `access_token`，未登录时为空字符串 `""` |
| `MD5` | — | 标准算法 | 32 位十六进制**小写** |
| `Base64` | — | 标准算法 | 对时间戳字符串的 UTF-8 字节 |

### 计算步骤

以 `path="/api/test"`, `timestamp=1700000000000`, `token="test_token"` 为例：

```
步骤 1: 时间戳 → 字符串
  ts_str = "1700000000000"

步骤 2: 内层 MD5
  inner = MD5("1700000000000" + "test_token")
        = MD5("1700000000000test_token")
        = "91a1f8f522e5247a824fe78bf88b62a0"

步骤 3: 拼接外层层字符串
  outer_str = "/api/test" + "?sign=" + "91a1f8f522e5247a824fe78bf88b62a0"
            = "/api/test?sign=91a1f8f522e5247a824fe78bf88b62a0"

步骤 4: 外层 MD5
  outer = MD5("/api/test?sign=91a1f8f522e5247a824fe78bf88b62a0")
        = "611caada2eb45d1c20e53b2fda90fc18"

步骤 5: Base64 时间戳
  b64 = Base64("1700000000000")
      = "MTcwMDAwMDAwMDAwMA=="

步骤 6: 拼装最终签名
  sign = "611caada2eb45d1c20e53b2fda90fc18" + "1." + "MTcwMDAwMDAwMDAwMA=="
       = "611caada2eb45d1c20e53b2fda90fc181.MTcwMDAwMDAwMDAwMA=="
```

> 注意 `"1."` 是字符串常量，不是版本号也不是其他含义。小程序源码中硬编码为 `"1."`。

### 签名在请求中的位置

```http
GET /api/flySource-yxgl/dormSignTask/getListForApp?current=1&size=10
Host: simp.csuft.edu.cn
Authorization: Basic Zmx5c291cmNl...
FlySource-Auth: eyJhbGciOiJSUzI1NiJ9...
FlySource-sign: 611caada2eb45d1c20e53b2fda90fc181.MTcwMDAwMDAwMDAwMA==
```

## 3. stuTaskId 校验哈希

打卡提交时，除了请求头中的 `FlySource-sign`，还需要在请求体中携带 `stuTaskId` 字段。

### 公式

```
stuTaskId = MD5(JSON.stringify({
    latitude: "...",
    longitude: "...",
    locationAccuracy: "...",
    signDate: "...",
    taskId: "...",
    fileId: "..."
}))
```

### Key 顺序要求

`JSON.stringify` 的输出中，**Key 的顺序必须与小程序源码中的对象字面量顺序完全一致**：

```
latitude → longitude → locationAccuracy → signDate → taskId → fileId
```

如果 Key 顺序不同，MD5 结果不同，服务器校验失败。

### Python 实现

```python
import json
from src.utils.crypto import md5

def _prepare_stu_sign_data(task_detail, lat, lng, accuracy, date, file_id, task_id):
    hash_input = {
        "latitude": str(lat),
        "longitude": str(lng),
        "locationAccuracy": accuracy,
        "signDate": date,
        "taskId": task_id,
        "fileId": file_id or "",
    }
    stu_task_id = md5(
        json.dumps(hash_input, ensure_ascii=False, separators=(",", ":"))
    )
    return {..., "stuTaskId": stu_task_id}
```

关键细节：
- `ensure_ascii=False` — 不转义非 ASCII 字符
- `separators=(",", ":")` — 紧凑格式，无空格（对齐小程序 `JSON.stringify` 默认输出）
- 坐标值转为字符串（`str(lat)`）

## 4. Basic Auth 认证头

### 公式

```
Authorization: Basic base64(ClientId + ":" + ClientSecret)
```

### 固定凭据

| 属性 | 值 |
|------|-----|
| ClientId | `flysource_wise_wxapp` |
| ClientSecret | `DA788asdUDjnasd_flysource_wxappdsdadDAIUiuwqe` |

均可通过环境变量覆盖：`FLYSOURCE_CLIENT_ID`、`FLYSOURCE_CLIENT_SECRET`。

### Python 实现

```python
def generate_basic_auth() -> str:
    raw = f"{CLIENT_ID}:{CLIENT_SECRET}"
    return "Basic " + b64_encode(raw.encode("utf-8"))
```

## 5. Python 实现

### 签名生成（`src/utils/sign.py`）

```python
from src.utils.crypto import md5, b64_encode

def generate_sign(path: str, timestamp: int, token: str = "") -> str:
    ts_str = str(timestamp)
    inner_hash = md5(ts_str + token)
    outer_str = path + "?sign=" + inner_hash
    outer_hash = md5(outer_str)
    return outer_hash + "1." + b64_encode(ts_str.encode("utf-8"))
```

### 底层加密函数（`src/utils/crypto.py`）

```python
import hashlib
import base64

def md5(data: str) -> str:
    return hashlib.md5(data.encode("utf-8")).hexdigest()

def b64_encode(data: bytes) -> str:
    return base64.b64encode(data).decode("utf-8")
```

### 调用位置（`src/core/client.py`）

```python
def _headers(self, url_path: str, ...) -> dict:
    ts = int(datetime.now(_BEIJING_TZ).timestamp() * 1000)
    h = {
        "Content-Type": "application/json",
        "charset": "utf-8",
        "Referer": self._referer,
        "User-Agent": self.WX_USER_AGENT,
    }
    h["Authorization"] = generate_basic_auth()
    if self.token:
        h["FlySource-Auth"] = self.token
        h["FlySource-sign"] = generate_sign(url_path, ts, self.token)
    return h
```

> 签名时间戳使用 **北京时间**（`_BEIJING_TZ = timezone(timedelta(hours=8))`），与用户手机上微信小程序的行为一致。

## 6. JavaScript 参考实现

### 签名生成（`scripts/sign.js`）

```javascript
const crypto = require('crypto');

function md5(s) {
    return crypto.createHash('md5').update(s, 'utf8').digest('hex');
}

function b64(str) {
    return Buffer.from(str, 'utf8').toString('base64');
}

function generateSign(path, timestamp, token) {
    const tsStr = String(timestamp);
    const inner = md5(tsStr + token);
    const outer = md5(path + '?sign=' + inner);
    return outer + '1.' + b64(tsStr);
}
```

### 命令行使用

```bash
node scripts/sign.js /api/test 1700000000000 test_token
# 输出: 611caada2eb45d1c20e53b2fda90fc181.MTcwMDAwMDAwMDAwMA==
```

## 7. 交叉验证

### 单步验证

```bash

const outer = md5('/api/test?sign=' + inner);
console.log('inner:', inner);
console.log('outer:', outer);
"

python -c "
from src.utils.crypto import md5
```

两者输出的 `inner` 和 `outer` 值应完全一致。

## 8. 时间戳与时区

### 时间戳生成

```python
from datetime import datetime, timezone, timedelta

BEIJING_TZ = timezone(timedelta(hours=8))
ts = int(datetime.now(BEIJING_TZ).timestamp() * 1000)
```

### 为什么用北京时间

微信小程序在手机上运行，`Date.now()` 返回手机本地时间的毫秒时间戳。中国用户手机时区为北京时间（UTC+8），所以签名时间戳应该反映北京时间。

`.timestamp()` 方法将 aware datetime 转换为 UTC epoch 秒数，所以：
- `datetime.now(BEIJING_TZ).timestamp()` 返回正确的 Unix 时间戳
- 北京时间 21:05 的 UTC epoch 值 = 世界协调时 13:05 的 UTC epoch 值

### GitHub Actions 时区陷阱

GitHub Actions `ubuntu-latest` 默认时区为 **UTC**。如果直接用 `datetime.now()`（无 tz），会拿到 UTC 时间，导致：
1. 签名时间戳偏差 8 小时
2. 打卡日期字段偏差 1 天（跨日期边界时）
3. 通知中的执行时间显示 UTC

**修复**：Python 代码中用 `datetime.now(_BEIJING_TZ)` 替代 `datetime.now()`，bash 脚本中 `export TZ='Asia/Shanghai'`。

## 9. 常见陷阱

### 🚨 陷阱 1：缺少微信环境头 → 误导"签名错误"

服务器在校验签名之前先检查 `Referer` 和 `User-Agent`。如果缺少这些头，直接返回 `{"msg": "签名错误"}`，**即使签名本身计算正确**。

必须携带的头：
```python
{
    "Referer": "https://servicewechat.com/wx0e47c34c9982aa09/7/page-frame.html",
    "User-Agent": "Mozilla/5.0 ... MicroMessenger ... MiniProgramEnv/android",
    "charset": "utf-8",
}
```

### 🚨 陷阱 2：签名路径不含 query string

```
正确: path = "/api/flySource-yxgl/dormSignTask/getListForApp"
错误: path = "/api/flySource-yxgl/dormSignTask/getListForApp?current=1&size=10"
```

### 🚨 陷阱 3：MD5 十六进制大小写

飞源服务器期望 **小写** 十六进制。Python 的 `hexdigest()` 和 Node.js 的 `digest('hex')` 默认都是小写，没问题。

### 🚨 陷阱 4：Base64 编码对象是时间戳字符串

```python
# 正确
b64_encode(str(timestamp).encode("utf-8"))

# 错误 — 对整数的字节表示做 Base64
b64_encode(timestamp.to_bytes(8, 'big'))
```

### 🚨 陷阱 5：stuTaskId 的 Key 顺序

`json.dumps()` 在 Python 3.7+ 中默认保持插入顺序，但必须确保字典的 Key 顺序与小程序源码一致。使用显式的字典字面量（不要从其他来源合并字典）。

### 🚨 陷阱 6：时区不一致

开发环境（Windows 北京时间）和 GitHub Actions（UTC）的 `datetime.now()` 返回值不同。务必使用 timezone-aware datetime：`datetime.now(timezone(timedelta(hours=8)))`。
